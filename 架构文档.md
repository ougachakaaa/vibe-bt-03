# vibe-bt-03 架构文档

## 目标

`vibe-bt-03` 是一个面向量化研究和历史回测的 Python 项目。当前项目已经具备 `database` 层，后续架构应围绕“数据进入、数据读取、研究、策略、回测、分析、交互入口”这条主线展开。

早期实现应保持轻量：先明确模块职责和依赖方向，再逐步填充具体功能，避免在项目变大后出现数据读取、研究逻辑、回测引擎和交互入口互相耦合的问题。

## 推荐目录结构

```text
src/vibe_bt_03/
  config/
  database/
  data_ingestion/
  data_access/
  core/
  quant/
  strategy/
  backtest/
  analytics/
  user_interface/
    cli/
    notebook/
```

## 模块职责

### config

负责项目配置、环境变量读取、常量管理和运行时设置。

适合放置：

- 数据库连接配置
- 数据源密钥和路径
- 默认交易日历、默认复权方式等项目常量
- 本地开发、测试、生产环境的配置差异

### database

负责数据库边界，目前项目中已经存在该模块。

适合放置：

- SQLAlchemy engine 和 session 管理
- ORM models
- repositories
- 面向上层调用的 readers 和 writers API
- 数据库健康检查和 schema 管理

原则：`database` 只处理持久化和查询边界，不承载量化研究、策略规则或回测撮合逻辑。

### data_ingestion

负责从外部来源获取数据、清洗数据、标准化字段，并写入数据库。

适合放置：

- CSV、Excel、Parquet 等本地文件读取
- 第三方行情接口适配器
- 股票、期货、交易日历、复权因子等数据导入流程
- 原始字段到项目内部输入类型的转换
- 批量入库任务

原则：`data_ingestion` 负责“把数据弄进来”，不负责研究和回测。

### data_access

负责为 `quant`、`strategy` 和 `backtest` 提供稳定的数据读取接口。

适合放置：

- 行情数据统一读取接口
- 交易日历读取接口
- 标的池读取接口
- 复权数据选择逻辑
- 面向回测的批量数据加载和缓存

原则：上层模块尽量通过 `data_access` 取数据，而不是直接依赖数据库模型或 repository。

### core

负责项目核心领域类型和协议，是多个业务模块共享的基础语言。

适合放置：

- Bar、Order、Trade、Position、Portfolio 等核心对象
- 时间周期、市场类型、订单方向、成交状态等枚举
- Strategy、Broker、DataPortal 等协议或抽象接口
- 通用领域错误类型

原则：`core` 应尽量稳定、轻依赖，不直接访问数据库或外部数据源。

### quant

负责正式的量化研究逻辑。

适合放置：

- 因子计算
- 信号生成
- 特征工程
- 截面分析和时间序列分析
- 研究用统计函数
- 研究实验的可复用组件

原则：`quant` 主要回答“什么信号可能有效”，不直接处理账户撮合、手续费、滑点等回测细节。

### strategy

负责把研究信号转换为可执行的策略规则。

适合放置：

- 选股规则
- 调仓规则
- 仓位目标生成
- 风控规则
- 策略参数定义

示例：某个动量因子属于 `quant`，但“每月调仓、买入排名前 20、等权配置、过滤停牌股票”属于 `strategy`。

### backtest

负责历史回测和仿真。

适合放置：

- 回测引擎
- 时间推进循环
- 订单生成和撮合
- 手续费、滑点、涨跌停、停牌处理
- 账户、现金、持仓、成交记录更新
- 单标的和组合级回测流程

原则：`backtest` 使用 `strategy` 产出的交易意图，并通过 `data_access` 获取历史数据。

### analytics

负责回测和研究结果的分析。

适合放置：

- 收益率、波动率、夏普、最大回撤等绩效指标
- 年度、月度、分标的表现统计
- 换手率、胜率、盈亏比等交易统计
- 归因分析
- 报告数据结构和图表数据准备

原则：`analytics` 负责解释结果，不负责运行回测。

### user_interface

负责用户交互入口。

子模块：

- `user_interface.cli`：命令行工具，例如初始化数据库、导入数据、运行回测、导出报告
- `user_interface.notebook`：Notebook 研究辅助函数，例如快速加载行情、展示回测结果、绘制图表

原则：交互层只负责编排调用，不直接写复杂业务逻辑。

## 推荐依赖方向

```text
user_interface
  -> config
  -> data_ingestion -> database
  -> quant / strategy -> data_access -> database
  -> backtest -> analytics
```

更细一点：

```text
data_ingestion -> database
data_access -> database
quant -> data_access, core
strategy -> quant, core
backtest -> strategy, data_access, core
analytics -> backtest, core
user_interface -> 以上业务模块
```

应避免：

- `database` 反向依赖 `quant`、`strategy` 或 `backtest`
- `core` 依赖数据库、外部 API 或 UI
- `backtest` 直接塞入大量报告绘图逻辑
- `user_interface` 中写入不可复用的业务规则

## 第一阶段建议

第一阶段可以先保持模块很薄，只做三件事：

1. 建立目录和 `__init__.py`，明确包边界。
2. 把已有 `database.api.readers` 通过 `data_access` 做一层更稳定的调用封装。
3. 在 `core` 中逐步沉淀回测所需的核心类型，例如订单、成交、持仓和账户。

等第一条简单策略跑通后，再扩展：

- `quant.factors`
- `strategy.rules`
- `backtest.engine`
- `analytics.metrics`
- `user_interface.cli.commands`

这样可以避免一开始过度设计，也能给后续扩展留下清晰位置。
